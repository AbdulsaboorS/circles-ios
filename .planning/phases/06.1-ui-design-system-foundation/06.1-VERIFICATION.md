| File                          | Line     | Pattern                                     | Severity | Impact                                                                                      |
|-------------------------------|----------|---------------------------------------------|----------|---------------------------------------------------------------------------------------------|
| `Circles/ContentView.swift`   | 13, 15   | `Color(hex: "0D1021")`, `.tint(Color(hex: "E8834B"))` in loading state | Warning  | Loading state uses raw hex instead of Color.appBackground / Color.accent — design system not fully adopted in this one branch, but ContentView.swift is listed as a phase 06.1 file |
| Multiple non-DesignSystem views | various | `Color(hex: "0D1021")`, `Color(hex: "E8834B")` throughout (CommunityView, CircleDetailView, etc.) | Info | Pre-existing views not in scope for 06.1 — these represent adoption work for future phases |
| `AppBackground.swift`         | 9, 13    | `Color(hex: ...)` inline instead of token aliases | Info | Explicitly permitted by Plan 02 spec: "AppBackground can also define them inline" — not a defect |
| `AppIconView.swift`           | 19, 22, 79 | `Color(hex: ...)` inline | Info | Canvas GraphicsContext does not use SwiftUI environment, inline values are correct here |

**Note on D-07 semantic aliases:** `Color.appBackground`, `Color.cardSurface`, etc. are implemented as dark-mode static defaults rather than adaptive resolvers. The plan text said they should "delegate to AppColors.resolve()" but the actual design is that callers use `AppColors.resolve(colorScheme)` for adaptive behavior, while the static aliases serve as dark-mode defaults. This matches the inline comment in DesignTokens.swift ("resolves to dark variant by default") and is consistent with how all DesignSystem components actually use these tokens (they call AppColors.resolve or read @Environment(\.colorScheme) directly). Not a bug.

### Human Verification Required

#### 1. Animated blob breathing in Simulator

**Test:** Run the app in Xcode Simulator, observe the home screen / any screen using AppBackground.
**Expected:** Two soft glowing ellipses at opposite corners slowly pulse (opacity + scale) in a 4-5 second rhythm.
**Why human:** Animation state cannot be verified by static code inspection.

#### 2. Sunrise/sunset auto dark mode

**Test:** Set `cityLatitude` / `cityLongitude` UserDefaults to a known location, then manually adjust device clock to just before/after sunrise.
**Expected:** App color scheme switches from dark to light at sunrise, light to dark at sunset without restart.
**Why human:** Timer-driven runtime behavior — requires Simulator time manipulation.

#### 3. ThemeMode manual override persistence

**Test:** Open settings or call `ThemeManager.shared.mode = .alwaysLight`, background the app, relaunch.
**Expected:** Mode persists and app launches in light mode.
**Why human:** UserDefaults persistence + app lifecycle requires runtime verification.

#### 4. AppIconView screenshot at 1024pt

**Test:** Open AppIconView Preview in Xcode, verify the rub el hizb tessellation fills the frame with amber stars on dark background.
**Expected:** A 4x4 grid of 8-pointed amber stars on #0E0B08, clipped to Apple icon corner radius.
**Why human:** Visual quality and geometric correctness require human eyes.
